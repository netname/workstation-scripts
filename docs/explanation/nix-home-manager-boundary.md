# Nix and Home Manager Boundary

Home Manager is excellent for files that should be generated from declarative configuration. Some application configs are better kept as normal editable files.

## Home Manager-Managed

Use Home Manager for:

- global CLI packages
- zsh setup
- git identity and aliases
- starship prompt
- direnv integration
- stable generated files

Apply with:

```bash
hms
```

Open a new shell when shell startup or PATH behavior changes.

## User-Managed

Keep files user-managed when the tool has its own live reload path or when direct editing is clearer:

- `wezterm/wezterm.lua`
- `tmux/tmux.conf`
- `scripts/sessionizer`
- `nvim/`
- VS Code recommendations

These files still live in `dotfiles`, but they are read directly by the application instead of being fully generated by Home Manager.

## Why the Split Exists

The split keeps rebuilds reproducible without making every edit go through Nix evaluation. Declarative config is used where it provides leverage; direct config files are used where they make daily iteration faster.
